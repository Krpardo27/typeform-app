# Resumen de cambios - 2026-07-14

Este documento resume los cambios implementados durante la jornada en el panel administrativo, la vista de workspaces y el flujo de Typeform.

## Objetivo general

Se ordeno la aplicacion para separar responsabilidades, mejorar la experiencia mobile, reforzar flujos sensibles y dejar mas trazabilidad en acciones administrativas.

## Admin

### Header reutilizable

- Se creo `AdminPageHeader` para evitar repetir headers en paginas del panel.
- Se integro en paginas de usuarios, workspaces, miembros y auditoria.
- Soporta `title`, `description`, `eyebrow`, `leading` y `actions`.
- Las acciones se acomodan responsivamente en mobile y desktop.

### Mobile dock administrativo

- Se adapto `AdminMobileDock` a las rutas reales del proyecto:
  - `/admin`
  - `/admin/workspaces`
  - `/admin/users`
  - `/admin/miembros`
  - `/admin/auditoria`
  - `/workspaces/me`
- Se monto en `app/admin/layout.tsx`.
- El sidebar queda para desktop y el dock se muestra en mobile.

### Auditoria

- Se descompuso `app/admin/auditoria/page.tsx`.
- Nuevos componentes:
  - `AdminAuditHeader`
  - `AuditStatsGrid`
  - `AuditTimeline`
- Se agrego `audit-timeline.service.ts` para normalizar eventos de auditoria y sesiones.
- El timeline ahora tiene overflow horizontal y vertical para manejar muchos registros.

### Miembros

- Se descompuso `app/admin/miembros/page.tsx`.
- Nuevos componentes:
  - `AdminMembersHeader`
  - `NoWorkspacesNotice`
  - `RecentWhitelist`
- Se agregaron confirmaciones con SweetAlert al autorizar miembros.
- Al autorizar un miembro, el usuario queda en estado `ACTIVE`.

### Usuarios

- Se agrego `DeleteUserButton` con confirmacion SweetAlert.
- Se agrego `deleteUserAction` para eliminar usuarios desde el panel.
- La eliminacion ahora revoca explicitamente:
  - sesiones
  - accesos a workspaces
  - whitelist (`AllowedUser`)
  - usuario
- Se evita eliminar la cuenta propia.
- Se evita eliminar al unico `SUPER_ADMIN`.

## Workspaces

### Sidebar y mobile dock

- Se creo `WorkspaceMobileDock` basado en `WorkspaceSidebar`.
- Incluye navegacion mobile para:
  - Panel
  - Inicio
  - Formularios
  - selector de radio/workspace
- `WorkspaceShell` ahora muestra sidebar en desktop y dock en mobile.
- Se agrego padding inferior al contenido para evitar que el dock tape la UI.

### Cards y responsividad

- Se ajustaron cards de workspaces con mejor soporte para nombres e IDs largos.
- Se mejoro la responsividad de vistas que antes dependian de grillas fijas.

## Typeform

### Ruta `/forms/new`

- Se corrigio el caso donde `/forms/new` podia entrar por la ruta dinamica `[formId]` y consultar `/forms/new` en Typeform.
- Se componentizo la pagina de crear formulario.
- Nuevos componentes:
  - `NewWorkspaceFormHeader`
  - `EmptyBaseFormsState`
  - `BaseFormsGrid`
  - `CreateDefaultFormButton`
- Se agrego confirmacion SweetAlert antes de crear formulario base.

### Ruta `/forms`

- Se reemplazo la tabla rigida por cards responsivas.
- Nuevos componentes:
  - `WorkspaceFormsHeader`
  - `EmptyWorkspaceFormsState`
  - `WorkspaceFormCard`
  - `WorkspaceFormsGrid`
- Las cards siguen el estilo de `WorkspaceCard`.
- Se mejoro soporte para titulos e IDs largos.

### Seleccion de ganadores

- Existe flujo para seleccionar uno o mas ganadores desde las respuestas de Typeform.
- La seleccion usa SweetAlert antes de confirmar.
- Se corrigio el submit del formulario usando `requestSubmit()` en vez de `form.submit()`.
- La visualizacion completa de datos sensibles quedo restringida al flujo de seleccion de ganadores.
- Se elimino la via global `showSensitive=1`.
- La cookie temporal de seleccion valida que pertenezca al usuario que hizo la accion.
- La cookie incluye:
  - tokens seleccionados
  - motivo
  - usuario actor
  - timestamp
- La auditoria `SENSITIVE_DATA_VIEWED` ahora se registra solo cuando realmente se revela data completa de ganadores.
- La auditoria incluye `scope: winner_selection_flow`.

## Autenticacion y OTP

- Se reforzo el envio y verificacion de OTP.
- `login.actions.ts` ahora es Server Action.
- Antes de enviar o verificar OTP se valida que:
  - el email este en whitelist
  - el usuario no este en estado `REJECTED`
- Si el email no esta autorizado, se eliminan OTP pendientes para ese email.
- Esto evita que un usuario eliminado o revocado pueda entrar usando un codigo viejo.

## Confirmaciones con SweetAlert

Se agregaron o corrigieron confirmaciones para acciones sensibles:

- Autorizar miembro
- Revocar miembro
- Crear workspace
- Actualizar permisos de usuario
- Eliminar usuario
- Crear formulario base
- Duplicar formulario
- Confirmar ganadores

## Auditoria sensible

- `WINNER_SELECTED` registra la seleccion de ganadores.
- `SENSITIVE_DATA_VIEWED` registra la visualizacion real de data completa.
- Se evito registrar visualizaciones sensibles cuando la informacion sigue enmascarada.
